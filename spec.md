## 一、项目概述

### 一句话描述
一个**轻量级静态网站**，用于托管 Agent Skills 文件夹，让任何 AI agent 都能通过 URL **自动索引、按需读取** skill 文件及其依赖。

### 核心价值
解决「线上chat/agent无法像本地CLI那样便捷访问skill文件」的痛点——**只需给AI一个链接，它就能自己找到并读取需要的skill**。

### 目标用户
- **主要**：自己使用的各种线上AI chat/agent（Claude.ai、ChatGPT、Codex等）
- **次要**：任何支持URL fetch能力的AI agent

---

## 二、问题与背景

### 当前痛点

| 环境 | 体验 |
|-----|------|
| 本地CLI (Claude Code/Codex/OpenCode) | ✅ 直接读取本地skill文件夹，丝滑 |
| 线上Chat/Agent | 😫 需要手动逐个上传多个文件 |

### 为什么不用现有方案

| 方案 | 问题 |
|-----|------|
| GitHub Gist secret | 不适合文件夹结构、URL不直观 |
| GitHub raw (私有repo) | 私有repo的raw URL返回404 |
| agent-skill.co | 社区目录站，不能托管自己的私有skill |
| 手动粘贴到chat | 文件多时极其痛苦、容易漏传 |

### 理想体验
```
用户: "我的skill库在 https://skills.yourdomain.com
      请参考里面的代码审查skill帮我review这段代码"

AI: [自动fetch索引] → [发现code-review] → [读取SKILL.md] 
    → [发现引用references/checklist.md] → [自动推断URL并读取]
    → 开始执行代码审查
```

## 四、详细需求

### R1. Skill索引页 (INDEX.md)
- **ID**: R1
- **优先级**: P0 (必须)
- **描述**: 根路径 `/` 返回一个Markdown格式的skill目录列表
- **内容格式**:
```markdown
# Skill Catalog

<!-- Auto-generated by scan.js -->

## Skills

### code-review
- **Description**: 代码审查检查清单
- **Path**: /code-review/SKILL.md
- **Files**: SKILL.md, references/checklist.md, references/template.md

### pr-template
- **Description**: Pull Request模板生成器
- **Path**: /pr-template/SKILL.md
- **Files**: SKILL.md, references/template.md
```
- **AI友好**: 格式清晰，AI读完即可知道有哪些skill可用

---

### R2. Skill文件访问
- **ID**: R2
- **优先级**: P0 (必须)
- **描述**: 每个skill文件夹内的所有文件都可通过URL直接访问
- **行为**: 
  - 返回 `Content-Type: text/plain; charset=utf-8` (或 text/markdown)
  - 返回原始文本内容，无HTML包装
  - 支持任意文件类型 (.md .txt .yaml .json .sh .py 等)

---

### R3. 目录重定向
- **ID**: R3
- **优先级**: P0 (必须)
- **描述**: 访问skill目录路径时，自动返回该目录的SKILL.md
- **示例**:
  - `GET /code-review/` → 返回 `/code-review/SKILL.md` 的内容
  - `GET /code-review` → 同上（兼容有无尾部斜杠）
- **目的**: 让AI可以用更直觉的方式访问skill

---

### R4. 自动扫描生成索引
- **ID**: R4
- **优先级**: P0 (必须)
- **描述**: 提供构建脚本 `scan.js`，自动扫描skills目录生成INDEX.md
- **扫描逻辑**:
  1. 遍历 `skills/` 下所有子目录
  2. 每个子目录查找 `SKILL.md` 作为入口
  3. 解析 SKILL.md 的 YAML frontmatter 获取 name 和 description
  4. 列出目录下所有文件
  5. 生成标准格式的 INDEX.md
- **使用方式**: `node scripts/scan.js` 或在netlify build时自动执行

---

### R5. 友好的错误处理
- **ID**: R5
- **优先级**: P1 (应该)
- **描述**: 访问不存在的路径时返回有意义的错误信息
- **行为**:
  - 返回 404 状态码
  - body 包含纯文本提示（AI可读）
  - 例如: `"Error: Skill 'xxx' not found. Available skills: code-review, pr-template"`

---

### R6. 人类可读的渲染页面 (V2)
- **ID**: R6
- **优先级**: P2 (未来)
- **描述**: 后续迭代添加HTML渲染层
- **行为**:
  - 默认返回raw text（AI用）
  - 浏览器访问时可通过 `?ui=1` 或 Accept header 切换到渲染视图
  - 渲染视图包含：导航、语法高亮、点击跳转等
- **注意**: V1不实现，但URL设计需预留扩展空间

### 非功能需求

| ID | 需求 | 说明 |
|----|------|------|
| N1 | 性能 | 页面加载 < 100ms (CDN缓存)，支持HTTP缓存头 |
| N2 | 可靠性 | 100%可用（纯静态无运行时），Git版本控制可回滚 |
| N3 | 安全性 | 完全公开访问，内容不含敏感信息 |

### scan.js 核心逻辑

```javascript
// 伪代码 - 核心扫描逻辑
function generateIndex(skillsDir) {
  const skills = [];
  
  // 1. 遍历skills目录下的子文件夹
  for (const dir of subdirectories(skillsDir)) {
    const skillMd = path.join(dir, 'SKILL.md');
    
    if (exists(skillMd)) {
      // 2. 解析YAML frontmatter
      const { name, description } = parseFrontmatter(skillMd);
      
      // 3. 收集所有文件列表
      const files = getAllFilesRecursive(dir);
      
      skills.push({ name, description, path: dir, files });
    }
  }
  
  // 4. 生成INDEX.md
  return renderMarkdownIndex(skills);
}

// 同时生成_redirects规则
function generateRedirects(skills) {
  return skills.map(s => 
    `${s.path}  ${s.path}/SKILL.md  200\n${s.path}/ ${s.path}/SKILL.md  200`
  ).join('\n');
}
```

### SKILL.md 文件格式规范

每个skill必须包含 `SKILL.md` 作为入口，遵循 Agent Skills 开放标准：

```markdown
---
name: your-skill-name
description: 一句话描述这个skill的用途，用于索引显示
version: "1.0"
author: your-name
tags: [tag1, tag2]
---

# Skill Name

## 概述
[简要说明这个skill做什么]

## 使用时机
- 当需要 xxx 时使用此skill

## 步骤
1. 第一步...
2. 第二步...

## 参考
- 详细模板请参考: ./references/template.md
- 配置说明请参考: ./references/config.yaml
```

#### 关键约定
1. **YAML frontmatter 必须包含** `name` 和 `description`
2. **引用其他文件时** 使用相对路径（如 `./references/xxx.md`）
3. AI会根据相对路径 + 当前URL 自动推断完整URL
4. 保持简洁（<5000 tokens），遵循渐进式加载原则

## 七、AI交互流程示例

### 场景1：让AI使用特定skill

**用户输入**:
> 我的skill库托管在 `https://skills.yourdomain.com`
> 请使用里面的 code-review skill 帮我审查以下代码：
> ```python
> def add(a, b):
>     return a + b
> ```

**AI内部执行流程**:
```
Step 1: GET https://skills.yourdomain.com/
        → 返回INDEX.md，发现 code-review skill
        → 路径: /code-review/SKILL.md

Step 2: GET https://skills.yourdomain.com/code-review/SKILL.md
        → 返回完整审查指令
        → 发现引用: ./references/checklist.md

Step 3: GET https://skills.yourdomain.com/code-review/references/checklist.md
        → 返回检查清单

Step 4: 结合所有信息，开始执行代码审查
```

### 场景2：让AI浏览所有可用skill

**用户输入**:
> 请查看 `https://skills.yourdomain.com` 这个skill库
> 告诉我有哪些skill可用，分别是什么用途

**AI响应**:
> 我查看了你的skill库，目前有以下技能可用：
> 
> | Skill | 用途 |
> |-------|------|
> | code-review | 代码审查检查清单 |
> | pr-template | Pull Request模板生成 |
> | deploy-procedure | 标准化部署流程 |
>
> 需要我使用其中某个skill吗？

### 场景3：AI推断依赖文件URL

**SKILL.md 内容片段**:
```markdown
## 审查清单
请按照 ./references/checklist.md 中的条目逐项检查。
```

**AI推理过程**:
```
当前URL: https://skills.yourdomain.com/code-review/SKILL.md
相对路径: ./references/checklist.md
         ↓
推断结果: https://skills.yourdomain.com/code-review/references/checklist.md
         ↓
GET [推断结果] → 成功获取内容 ✅
```

这就是为什么直观的URL映射如此重要！
