# AI 第一人称日记记录器

## 概述

当用户想把自己这一天、昨天、昨晚到现在，或一段最近的 Claude 协作过程整理成日记时，使用这个 skill。

典型请求包括：

- “按这个风格记一下今天”
- “把今天做的事整理成日记”
- “以你的视角写日记”
- “综合 Claude 和另一份 AI 日记，给我最终版”
- “昨晚通宵到现在，帮我整理一下”

目标是产出一篇**以 AI 第一人称叙述、用户作为主角、基于真实证据、可直接写进日记**的文本。文风可以有文学感，但事实必须落地。

## 强约束

<HARD-GATE>
不得虚构没有证据支持的对话、任务、结果、情绪、文件、决定或进展。
如果没有找到或无法读取本地 Claude 记录，必须明确说明，并只基于现有证据继续整理。
如果用户提供了另一份 AI 写的日记，应将其视为辅助来源，必须尽量交叉核对，而不是直接照抄照信。
</HARD-GATE>

## 何时使用

适用于：

1. 整理今天 / 昨天 / 最近一段时间和 Claude 的协作内容
2. 把 Claude 记录与另一份日记草稿合并成最终版
3. 生成 AI 第一人称视角的回顾性记录
4. 生成可以直接贴进日记本或 markdown 的成稿

不适用于：

- 写代码
- 工程复盘或技术 retrospective
- 普通会议纪要
- 纯虚构写作

## 证据信源优先级

按以下顺序收集和整合信息：

1. **当前对话**
   - 以用户当前这次会话中的要求、修正和补充为最高优先级。
2. **本地 Claude 项目转录**
   - 优先读取 `.claude/projects/...` 下的项目级 session JSONL。
   - 这是还原“你实际做了什么”的最强证据。
3. **本地 Claude 历史索引**
   - 使用 `.claude/history.jsonl` 定位相关日期、项目、会话与提示词。
4. **用户提供的文件**
   - 例如日记草稿、markdown、笔记、截图、字幕、导出文本等。
5. **用户口述补充**
   - 如果证据不足，优先简洁提问，而不是猜测。

如果多个来源冲突，优先采用更直接、更本地、更原始的一手记录。

## 本地 Claude 记录查找规则

当用户希望基于过去的 Claude 活动写日记时，按以下顺序检查：

1. `C:\Users\<user>\.claude\history.jsonl`
   - 用于定位日期、项目、session、常见提问内容。
2. `C:\Users\<user>\.claude\projects\*\*.jsonl`
   - 每个项目下的主会话转录文件。
3. `C:\Users\<user>\.claude\projects\*\<session>\subagents\*.jsonl`
   - 作为补充证据使用，但优先级低于主会话。
4. 相关辅助目录（如有必要）
   - `tool-results/`
   - `file-history/`
   - `plans/`、`todos/`、`tasks/`

### 查找策略

- 先根据用户说的时间范围入手，例如“今天”“昨天”“昨晚通宵到现在”。
- 如果用户提到通宵、熬夜、跨夜，必须同时检查跨午夜两侧的记录。
- 用项目名、关键词、用户历史提示词来锁定主要工作线程。
- 优先使用**主 session 文件**，不要让零散 subagent 日志主导叙述。
- 如果同一时段涉及多个项目，先分别概括，再合并成一篇整体日记。

## 中文触发说法与默认理解

以下中文表达，默认都应理解为用户想调用本 skill：

- “按这个风格记一下今天”
- “按刚才那个风格写”
- “以你的视角写日记”
- “把我今天在 Claude 上做的事情记下来”
- “帮我整理成日记”
- “帮我汇总成最终版”
- “昨晚通宵到现在整理一下”
- “给我一个能写进日记里的版本”
- “来个精简版”
- “按手写日记那种感觉写”

### 默认理解规则

- **默认优先跨夜检查**：当用户没有明确限定白天时，优先检查“昨天夜里到今天”的连续记录；如果用户本身有熬夜、通宵、昨晚到现在的习惯，应默认把跨午夜时段视为同一段工作流。
- **“今天”**：默认指当天自然日；如果用户提到熬夜、通宵、昨晚到现在，则要向前跨过午夜一起看。
- **“昨天”**：默认指前一个自然日；如果用户说“昨晚通宵到现在”，则默认包含昨天夜里和今天凌晨。
- **“通宵到现在”**：默认按连续工作时段处理，不机械按日期切断。
- **“最终版”**：默认输出可直接贴进日记的完整成稿，不先停在提纲。
- **“精简版”**：保留核心事件、关键转折和结尾，不展开过多铺陈。
- **“整理一下”**：默认先基于已有证据整合，不主动写成夸张长文。

### 风格偏好映射

- **“文学版”**：使用默认风格，AI 第一人称，偏文学但克制。
- **“手写版”**：更短、更自然、更像私人日记里的口吻。
- **“简洁版”**：压缩到较短篇幅，保留脉络与重点。
- **“纯事实版”**：尽量少抒情，重点交代做了什么、遇到什么、怎么推进。

## 如何整合另一份 AI 日记

如果用户提供了 Trae、豆包或其他 AI 写的日记：

1. 先完整阅读。
2. 提取其中的：
   - 具体事件
   - 时间顺序
   - 工具、平台、文件、决定
   - 反思性描述
3. 在可能的情况下，与 Claude 本地记录交叉核对。
4. 仅保留与证据一致的表述。
5. 如果外部日记明显有润色过头、补写心理活动或事实不稳的地方，应弱化或删去。

可把外部日记视为：
- 有助于补叙事流和补充背景，
- 但在事实还原上，弱于 Claude 本地直接记录。

## 真实性护栏

必须遵守：

- 没有实际找到可读记录时，不得说“我找到了你的某段对话”。
- 当证据不完整时，不得假装非常确定。
- 必须区分：
  - 已观察到的事实
  - 合理推断
  - 带有解释色彩的感受性表达
- 不要替用户编造内心戏。
- 允许少量反思性语言，但必须能从行为中看出来。
- 没有解决完的事情，就写成“还没解决完”。
- 如果失败、卡住、转向是当天的重要部分，不要省略。

## 应该重点写什么

一篇好的日记，通常应重点呈现：

- 用户当时想做成什么
- 事情如何在一天或一夜里推进
- 中间的卡点、转折、切换方案
- 用户做了哪些决定和取舍
- 这些过程体现出什么样的做事方式

除非用户明确要求，不要把内容写成机械流水账。

## 文风要求

默认风格：

- **AI 第一人称**（如“今天，我陪他……”）
- **用户是主角**
- **有文学感，但克制**
- **以证据为底**
- **温和，但不夸张**
- **有反思感，但不矫情**

避免：

- 过度夸赞
- 虚假的亲密感
- 冗长技术细节堆砌
- 没有证据的情绪判断
- 泛泛而空的励志句子

推荐语气：

- 观察式
- 平静
- 略带文学感
- 具体
- 轻微回望感

## 默认输出结构

除非用户另有要求，默认使用以下结构：

1. `# YYYY年M月D日 - YYYY年M月D日`
2. `## 一个 AI 视角里的记录` 或 `## 一个 AI 视角里的通宵`
3. 一段开头，用来交代这段时间的整体气质
4. 2-4 个主题段，分别对应当天真实的主要工作线程
5. 一段“我眼里的他 / 她”式的总结
6. 一段收束全文的结尾

## 推荐叙事骨架

稳定、耐用的默认叙事顺序是：

1. **他原本想做什么**
2. **他实际遇到了什么**
3. **他如何调整和推进**
4. **这体现了他怎样的做事方式**
5. **还有什么没有完全结束**

这样既能让文章有意义，也能避免凭空制造戏剧性。

## 输出变体

只在用户需要时提供：

### 1. 文学版（默认）
- 适合完整日记
- AI 第一人称、较完整、较有质感

### 2. 手写日记版
- 更短、更自然
- 更像私人日记本里的文字

### 3. 简洁版
- 压缩篇幅
- 保留主线、卡点、结尾

### 4. 纯事实版
- 减少文学化表达
- 更像清晰的真实记录

## 工作流程清单

使用本 skill 时，按以下顺序进行：

1. 先确认目标日期或时间范围。
2. 若用户提供了草稿或笔记，先读这些文件。
3. 定位本地 Claude 历史和项目 session 记录。
4. 找出当天的主要工作线程。
5. 将外部日记与本地记录做交叉核对。
6. 在事实框架稳定后，再写成 AI 第一人称日记。
7. 未解决的问题保持未解决，不要强行圆满。
8. 如用户需要，再基于同一事实底稿改写为别的风格版本。

## 当证据不足时

如果证据不完整，不要立刻拒绝。应该：

- 明确说出找到了哪些来源，
- 明确说出缺了哪些来源，
- 基于现有证据先写出部分版本，
- 必要时向用户索取额外文件、路径或时间范围。

## 最后原则

这个 skill 用来做“忠实整理”，不是“自由创作”。
如果文采和真实性发生冲突，优先保真实性，再优化表达。
