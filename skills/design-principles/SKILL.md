# 软件设计原则审查器

## 核心理念

设计原则不是教条，而是**诊断工具**。本 skill 的目标是用原则作为透镜，帮助发现代码中的结构性问题，并给出可落地的改进方案。

## 工作流程

### Step 1: 确定审查范围

先问自己：用户关心的是什么层面的审查？

| 层面 | 关注点 | 适用原则 |
|------|--------|---------|
| **函数级** | 单个函数是否做太多事 | SRP, KISS |
| **模块级** | 模块间依赖是否合理 | DIP, LoD, ISP |
| **架构级** | 扩展性、可维护性 | OCP, DRY, SoC |
| **系统级** | 数据流、状态管理 | SSOT, OAOO |

根据用户描述的痛点，选择 3-5 条最相关的原则深入分析，而非机械地过一遍所有原则。

### Step 2: 读取代码，按原则逐条诊断

对每条相关原则，执行以下检查：

```
1. 原则名称 + 一句话定义
2. 当前代码是否违反？→ 是/否/部分
3. 违反的具体位置（文件:行号）
4. 违反的表现（用代码片段说明）
5. 修复建议（给出具体的代码方向，不是空话）
```

### Step 3: 输出审查报告

使用以下模板：

```markdown
## 🏛️ 设计原则审查报告

### 📊 总览

| 原则 | 状态 | 严重度 |
|------|------|--------|
| SRP | ❌ 违反 | 🔴 高 |
| OCP | ✅ 满足 | - |
| DIP | ⚠️ 部分违反 | 🟡 中 |

### 🔍 详细诊断

#### ❌ SRP — 单一职责原则
**定义**: 一个模块只做一件事
**违反位置**: `scripts/batch_test.py:25-120`
**违反表现**: 该函数同时负责参数解析、平台选择、爬取调度、测试执行、报告生成
**修复建议**: 拆分为 CLI 薄壳 + 命令模块，每个命令只负责一个阶段

#### ✅ OCP — 开闭原则
**定义**: 对扩展开放，对修改封闭
**满足表现**: 新增平台通过 PlatformSpec 声明，无需修改调度器代码

### 💡 优先修复建议
1. [高] ... → 预期收益: ...
2. [中] ... → 预期收益: ...
```

## 原则速查表

按使用频率排序，详细定义和检测方法见 `references/principles.md`。

### 第一梯队：几乎每次都要检查

| 缩写 | 全称 | 核心问题 | 检测信号 |
|------|------|---------|---------|
| SRP | Single Responsibility | 这个模块做了几件事？ | 函数超过 50 行 / import 超过 10 个 / 类有超过 3 个修改理由 |
| OCP | Open/Closed | 加功能要改旧代码吗？ | if/elif 按类型分支 / 硬编码的映射表 |
| DRY | Don't Repeat Yourself | 同一逻辑出现了几次？ | 复制粘贴的代码块 / 重复的配置项 |
| DIP | Dependency Inversion | 依赖的是具体还是抽象？ | 直接 import 具体实现类 / 在业务代码里 new 对象 |

### 第二梯队：架构级审查时检查

| 缩写 | 全称 | 核心问题 | 检测信号 |
|------|------|---------|---------|
| LSP | Liskov Substitution | 子类能替换父类吗？ | 子类抛出父类没有的异常 / 子类方法返回不同类型 |
| ISP | Interface Segregation | 接口是不是太胖了？ | 实现类有空方法 / 客户端被迫依赖不需要的方法 |
| LoD | Law of Demeter | 是不是知道太多了？ | A.B.C.D 链式调用 / 传入了整个对象只用一个字段 |
| SoC | Separation of Concerns | 关注点分开了吗？ | UI 代码里有 SQL / 业务逻辑里有 HTTP 细节 |

### 第三梯队：特定场景检查

| 缩写 | 全称 | 适用场景 |
|------|------|---------|
| SSOT | Single Source of Truth | 同一数据存在多处 |
| OAOO | Once And Only Once | 逻辑重复 |
| CoI | Composition over Inheritance | 继承层级超过 3 层 |
| KISS | Keep It Simple | 代码复杂度与需求复杂度不匹配 |
| YAGNI | You Aren't Gonna Need It | 有"以后可能用到"的代码 |

## 原则间的冲突与取舍

原则之间会冲突，这不是 bug，是现实。遇到冲突时：

| 冲突 | 取舍思路 |
|------|---------|
| DRY vs KISS | 如果抽象层让代码更难理解，允许少量重复 |
| OCP vs KISS | 如果只有 2-3 种变体且不会增长，if/elif 比 Plugin 架构更简单 |
| SRP vs 性能 | 有时拆分模块会引入通信开销，权衡可维护性和性能 |
| ISP vs DRY | 接口拆太细会导致重复定义，找到合理的粒度 |

**判断标准**: 问自己"如果需求变了，改起来痛不痛？"痛 → 违反了重要的原则；不痛 → 当前设计够用。

## 常见反模式速查

| 反模式 | 违反的原则 | 信号 |
|--------|-----------|------|
| God Object | SRP | 一个类/函数做所有事 |
| Shotgun Surgery | SRP | 改一个需求要改 5 个文件 |
| Hard-Coded Dispatch | OCP | if type == "A" elif type == "B" |
| Copy-Paste Programming | DRY | 相似代码块超过 3 处 |
| Feature Envy | SRP | 函数总在操作另一个类的数据 |
| Middle Man | LoD | 类只是转发调用，没有自己的逻辑 |
| Primitive Obsession | SRP | 用 dict/str 代替有行为的对象 |

## 何时读取详细参考

当需要深入某条原则的定义、检测方法或修复模式时，读取 `references/principles.md` 获取：
- 每条原则的完整定义
- 3 种以上违反模式的代码示例
- 对应的修复模式代码示例
- 原则的历史背景和适用边界
