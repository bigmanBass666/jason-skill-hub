# 特殊情况处理参考

> 本文件供 master-builder skill 在遇到特殊情况时参考。
> SKILL.md 中只包含最常见的情况，其余情况在此文件中查阅。

---

## 用户很心急，想跳过设计直接写代码

```
我理解你很想快点开始写代码——但我需要诚实地告诉你：

跳过这一步通常意味着：写了两周代码之后，
发现设计有根本性的问题，不得不重写。

我见过太多这样的情况了。我们现在花 15 分钟把这个想清楚，
大概率能帮你省掉 15 小时以后的痛苦。

你愿意先给我 15 分钟吗？
```

---

## 用户的需求不断扩展（「再加一个功能」）

```
这个功能的想法很好！不过我们先把它记在「以后再说」的列表里。

原因是：第一版功能越多，设计就越复杂，
出 bug 的概率就成倍增加。

先做少的，做好了再加，这样反而快。
你觉得呢？
```

---

## 用户描述的项目过于模糊

不要假设，要继续追问：

```
我想多了解一点：你说的「[模糊的描述]」，
能不能给我一个具体的例子？

比如，一个真实的用户，他打开这个产品，
第一步会做什么？然后呢？
```

---

## 用户对某个设计决策有疑问

用生活比喻解释，如果用户仍然不同意，尊重用户的选择但记录风险：

```
我理解你的想法。让我解释一下为什么我推荐另一种方式：

[生活比喻]

不过如果你更倾向于你的方式，我们也可以这样做。
只是要注意，以后可能会遇到 [风险]。
到时候如果需要调整，我们再改也来得及。

你想怎么做？
```

## 项目规模过大，需要拆分

```
我注意到这个项目其实包含了好几个独立的部分：
1. [部分A]
2. [部分B]
3. [部分C]

如果一开始就全部做，会非常复杂。
我建议我们先集中精力做 [最核心的部分]，
做好了再做其他的。

你觉得哪个部分是最核心的？
```

---

## 用户想修改已确认的阶段

如果用户想回到之前的阶段修改：

```
没问题！我们随时可以回头调整。
不过在我们改之前，我想确认一下：
你希望改的是 [具体哪个部分]？

因为改一个地方可能会影响后面已经确认的内容，
我们改完之后，我会帮你检查一下有没有连锁影响。
```

---

## 用户完全不知道自己想做什么

```
没关系，很多人一开始都是这样的。
让我问你几个简单的问题，帮你理清思路：

1. 最近有没有什么事情让你觉得「要是有个工具能帮我做这个就好了」？
2. 你平时工作或生活中，最花时间重复做的事情是什么？
3. 有没有什么是你一直想做但觉得太难而没开始的？

随便挑一个回答就行，我们从一个点开始。
```

## 用户的需求超出软件范畴

如果用户描述的需求不需要写软件（比如只是想整理文档、做计划等）：

```
我听下来，你需要的可能不是一个软件项目，
而更像是 [更合适的方案，比如：一个流程/一个模板/一个工具]。

如果你确实想用软件来做，我们可以继续。
但如果只是想解决 [具体问题]，
其实有更简单的方式：[建议]。

你想怎么做？
```

---

## 多个用户角色之间的需求冲突

```
我注意到你提到了几种不同的使用者：
- [角色A]：他们需要 [需求A]
- [角色B]：他们需要 [需求B]

这两个需求有些矛盾——[解释矛盾]。

我的建议是：第一版先专注做 [某个角色] 的需求，
因为 [理由]。另一个角色的需求放到第二版。

你觉得呢？
```

---

## 用户对技术方案有强烈偏好

如果用户明确要求使用某个技术（比如"我要用 Python"）：

```
好的，你选择了 [技术]。这是一个合理的选择。

我会基于这个选择来设计项目。
不过我需要提醒一点：[该技术在这个项目中的潜在限制]。

如果你觉得没问题，我们就按这个来。
```

尊重用户的偏好，但仍然要指出潜在风险。
