# 项目模板参考

> 本文件供 master-builder skill 在 Phase 2 自动判断项目规模和选择结构时参考。
> 根据用户在 Phase 1 描述的需求，选择最匹配的模板。

---

## 模板 A：个人工具 / 脚本 / 原型

**适用场景**：就自己用、功能单一、快速验证想法

**推荐结构**（2-3 层）：

```
┌──────────────────────────┐
│  👀 入口                  │  ← 命令行 / 简单界面 / API 调用
├──────────────────────────┤
│  🧠 核心逻辑              │  ← 做事的代码
├──────────────────────────┤
│  💾 数据（可选）           │  ← 配置文件 / 本地存储
└──────────────────────────┘
```

**生活比喻**：一个小工作室——工作台 + 工具箱，够用就好。

**设计要点**：
- 一个入口文件 + 一个核心模块即可
- 如果需要存数据，加一个简单的数据层
- 不要过度设计，保持简单

**常见项目类型**：
- 命令行工具
- 数据处理脚本
- 自动化脚本
- API 调用封装
- 个人效率工具

## 模板 B：小型应用 / AI 助手

**适用场景**：有交互界面、需要记忆/状态、功能 3-5 个

**推荐结构**（4 层）：

```
┌─────────────────────────────────┐
│  👀 用户看到的（门面）            │  ← 网页 / 聊天窗口 / 操作界面
├─────────────────────────────────┤
│  🧠 做决策的（大脑）              │  ← 决定要做什么、先做什么、做完了吗
├─────────────────────────────────┤
│  🔧 会用的工具（工具箱）           │  ← 能力：搜索/写文件/发邮件/查数据等
├─────────────────────────────────┤
│  💾 记东西的（仓库）              │  ← 记住：对话历史/用户数据/中间结果
└─────────────────────────────────┘
```

**生活比喻**：一栋小楼——门厅接待、办公室做事、仓库存东西、地下室放设备。

**设计要点**：
- 门面层只管展示和接收输入，不做业务逻辑
- 大脑层是核心，决定调用哪些工具、怎么组合
- 工具层每个工具独立，互不依赖
- 仓库层统一管理所有数据，其他层通过仓库层读写

**常见项目类型**：
- AI 聊天助手
- 任务管理应用
- 内容生成工具
- 数据分析仪表盘
- 小型 SaaS 应用

## 模板 C：多人协作 / 对外服务

**适用场景**：多人使用、需要用户系统、功能模块多、对外提供服务

**推荐结构**（模块化分层 + 清晰边界）：

```
┌──────────────────────────────────────────┐
│  👀 多个入口（门面群）                      │
│  ├── 网页端                                │
│  ├── 手机端                                │
│  └── API 接口                              │
├──────────────────────────────────────────┤
│  🧠 业务服务层（各部门）                     │
│  ├── 用户服务                              │
│  ├── 核心业务服务                           │
│  └── 通知服务                              │
├──────────────────────────────────────────┤
│  🔧 共享工具层（公共设施）                    │
│  ├── 权限检查                              │
│  ├── 日志记录                              │
│  └── 消息队列                              │
├──────────────────────────────────────────┤
│  💾 数据层（档案室）                         │
│  ├── 用户数据库                             │
│  ├── 业务数据库                             │
│  └── 缓存                                  │
├──────────────────────────────────────────┤
│  🌐 外部世界（外面的世界）                    │
│  ├── 第三方服务                             │
│  ├── 文件存储                               │
│  └── 邮件/短信                              │
└──────────────────────────────────────────┘
```

**生活比喻**：一个园区——多栋楼各有功能，道路相连，统一管理。

**设计要点**：
- 每个服务独立部署、独立开发
- 服务之间通过明确的约定通信
- 共享工具层提供横切关注点（权限、日志等）
- 数据层每个服务有自己的数据存储，不共享数据库
- 入口层可以对接多种客户端

**常见项目类型**：
- 电商平台
- 社交应用
- 企业管理系统
- 开放 API 平台
- 多租户 SaaS

## 模板选择决策树

```
用户描述的需求
│
├── 只有自己用？
│   └── 是 → 模板 A
│
├── 有交互界面？需要记住东西？
│   └── 是 → 模板 B
│
├── 多人使用？需要用户系统？
│   └── 是 → 模板 C
│
└── 不确定？
    └── 从模板 B 开始，比 A 多不了多少，但扩展性更好
```

**重要原则**：宁可从简单模板开始再升级，也不要一开始就选复杂模板。从 B 升级到 C 比从 A 直接跳到 C 容易得多。
