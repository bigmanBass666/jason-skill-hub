# 设计原则参考（内部使用）

> 本文件供 master-builder skill 内部使用，不直接向用户展示原则名称和技术定义。
> 在 Phase 4 质量守护阶段，按此文件逐条审查蓝图。

---

## 第一梯队：每次都必须检查

### SRP — 单一职责原则

**生活比喻**：一个房间只做一件事。厨房做饭，卧室睡觉，客厅待客。如果一个房间又做饭又睡觉又待客，迟早会乱。

**核心问题**：这个模块做了几件事？

**检测信号**：
- 函数超过 50 行
- 一个类的 import 超过 10 个
- 一个类有超过 3 个修改的理由
- 模块描述里出现"和"字（"负责A和B"）

**融入时机**：
- Phase 2 画分层结构时：每层只做一件事
- Phase 3 定义接口时：每个接口只承诺一类功能
- Phase 4 审查时：检查是否有模块职责过多

**推荐做法**：
- 拆分：一个模块只做一个"主题"的事
- 命名：如果模块名需要"和"来连接，说明该拆了
- 测试：如果你需要用一段很长的话来描述一个模块做什么，它可能做了太多

---

### OCP — 开闭原则

**生活比喻**：加新房间不用拆旧墙。预留好门和走廊，以后想加新房间直接接上去。

**核心问题**：加功能要改旧代码吗？

**检测信号**：
- if/elif 按类型分支
- 硬编码的映射表
- 加一个新类型需要改多处代码
- switch-case 按类型分发

**融入时机**：
- Phase 2 设计模块关系时：预留扩展点
- Phase 3 定义接口时：用接口而非具体实现
- Phase 4 审查时：检查加新功能是否需要改旧代码

**推荐做法**：
- 用注册/插件机制代替 if/elif
- 定义扩展接口，新功能实现接口即可接入
- 将变化的部分抽象出来，稳定的部分封闭

---

### DRY — 不重复原则

**生活比喻**：同样的东西只做一次。如果三个房间都需要热水，你不会在每个房间装一个热水器，而是装一个中央热水系统。

**核心问题**：同一逻辑出现了几次？

**检测信号**：
- 复制粘贴的代码块
- 重复的配置项
- 相似的函数只有个别参数不同
- 同一个规则在多处定义

**融入时机**：
- Phase 2 设计数据流时：统一处理路径
- Phase 3 定义接口时：公共逻辑提取为共享模块
- Phase 4 审查时：检查是否有重复逻辑

**推荐做法**：
- 提取公共函数/模块
- 用配置代替硬编码的重复
- 用模板方法模式消除相似逻辑的重复

---

### DIP — 依赖倒置原则

**生活比喻**：依赖承诺，不依赖具体的人。插座不关心电从哪个发电站来，只要电压对就行。灯泡依赖"插座"这个约定，不依赖"某某发电站"。

**核心问题**：依赖的是具体还是抽象？

**检测信号**：
- 直接 import 具体实现类
- 在业务代码里 new 对象
- 模块之间直接引用对方的内部细节
- 测试时无法替换依赖

**融入时机**：
- Phase 2 设计模块关系时：模块之间通过约定通信
- Phase 3 定义接口时：先定义接口，再定义实现
- Phase 4 审查时：检查模块间是否直接依赖具体实现

**推荐做法**：
- 定义接口/协议，模块之间只依赖接口
- 用依赖注入传入具体实现
- 高层模块不 import 低层模块的具体类

## 第二梯队：架构级审查时检查

### SSOT — 单一真相源

**生活比喻**：公司里只有一个人事部管员工信息。不会财务部管一份、行政部也管一份，否则两边数据不一致就乱套了。

**核心问题**：同一数据存在多处吗？

**检测信号**：
- 同一份数据在多个模块各存一份
- 更新数据需要改多个地方
- 不同模块读到的同一数据值不同

**融入时机**：
- Phase 2 设计记忆层时：统一数据存储
- Phase 3 定义接口时：数据只通过一个接口读写
- Phase 4 审查时：检查数据是否有多个来源

**推荐做法**：
- 每类数据只有一个权威来源
- 其他模块通过接口获取，不自己存
- 需要缓存时，明确缓存策略和失效机制

---

### SoC — 关注点分离

**生活比喻**：做饭的人不管洗碗，洗碗的人不管买菜。各司其职，互不干扰。

**核心问题**：不同的事分开了吗？

**检测信号**：
- 界面代码里有数据存储细节
- 业务逻辑里有界面展示细节
- 一个函数既处理数据又处理展示
- 模块描述里出现"并且"（"负责A并且B"）

**融入时机**：
- Phase 2 画分层结构时：每层只管自己的事
- Phase 3 定义接口时：接口只暴露该层关注的内容
- Phase 4 审查时：检查是否有关注点混杂

**推荐做法**：
- 严格分层，每层只做自己的事
- 界面层只管展示，不管数据怎么来
- 数据层只管存取，不管怎么展示
- 业务层只管逻辑，不管展示和存储的细节

---

### LoD — 迪米特法则（最少知道原则）

**生活比喻**：你点外卖不需要知道厨房里几个厨师、用什么锅。你只需要知道"下单→收到饭"。

**核心问题**：是不是知道太多了？

**检测信号**：
- A.B.C.D 链式调用
- 传入了整个对象只用一个字段
- 模块需要了解另一个模块的内部结构

**融入时机**：
- Phase 3 定义接口时：接口只暴露必要信息
- Phase 4 审查时：检查是否有过度耦合

**推荐做法**：
- 只跟直接朋友说话
- 接口参数尽量用简单类型或专用数据类
- 不要暴露内部结构给外部

---

### ISP — 接口隔离原则

**生活比喻**：你不会把电视的电源、信号、遥控全部接到一个插头上。分开接口，各管各的。

**核心问题**：接口是不是太胖了？

**检测信号**：
- 实现类有空方法
- 客户端被迫依赖不需要的方法
- 一个接口超过 7 个方法

**融入时机**：
- Phase 3 定义接口时：按使用者拆分接口
- Phase 4 审查时：检查接口是否过于臃肿

**推荐做法**：
- 按使用者角色拆分接口
- 每个接口只包含使用者需要的方法
- 大接口拆成多个小接口

## 第三梯队：特定场景检查

### LSP — 里氏替换原则

**生活比喻**：你买了一个"灯泡"，不管是什么牌子的灯泡，拧进去就应该能亮。如果某个牌子的灯泡拧进去会爆炸，那它就不算"灯泡"。

**适用场景**：使用继承时

**检测信号**：
- 子类抛出父类没有的异常
- 子类方法返回不同类型
- 子类重写方法但行为完全不同
- 使用子类时需要 isinstance 检查

**推荐做法**：
- 子类必须能完全替代父类使用
- 如果不能替代，考虑用组合代替继承

---

### KISS — 保持简单

**生活比喻**：能用螺丝刀拧的螺丝，不要用电动工具。杀鸡不用牛刀。

**适用场景**：代码复杂度与需求复杂度不匹配时

**检测信号**：
- 简单需求用了复杂设计模式
- 代码比需求描述还难理解
- 过度抽象

**推荐做法**：
- 先用最简单的方式实现
- 只有当简单方式不够用时才增加复杂度
- 如果 3 种变体以内且不会增长，if/elif 比插件架构更简单

---

### YAGNI — 你不会需要它

**生活比喻**：你不会在搬家前就买好新房子里每个房间的家具。先搬进去，看需要什么再买。

**适用场景**：发现"以后可能用到"的代码时

**检测信号**：
- 有"以后可能用到"的代码
- 为假设的未来需求做了设计
- 功能比需求多

**推荐做法**：
- 只做当前确定需要的功能
- 预留扩展点（OCP），但不提前实现
- 如果不确定是否需要，就不做

---

### CoI — 组合优于继承

**生活比喻**：你不需要成为厨师才能做饭，你可以请一个厨师来帮忙。组合就是"请人帮忙"，继承就是"自己成为那个人"。

**适用场景**：继承层级超过 3 层时

**检测信号**：
- 继承层级超过 3 层
- 子类只是为了复用代码而继承
- 继承关系不符合"是一个"的关系

**推荐做法**：
- 优先用组合（has-a）而非继承（is-a）
- 通过接口实现多态，而非继承
- 只在真正的"是一个"关系时使用继承

## 原则冲突与取舍

原则之间会冲突，这是现实。遇到冲突时：

| 冲突 | 取舍思路 |
|------|---------|
| DRY vs KISS | 如果抽象层让代码更难理解，允许少量重复 |
| OCP vs KISS | 如果只有 2-3 种变体且不会增长，if/elif 比插件架构更简单 |
| SRP vs 性能 | 有时拆分模块会引入通信开销，权衡可维护性和性能 |
| ISP vs DRY | 接口拆太细会导致重复定义，找到合理的粒度 |
| YAGNI vs OCP | 预留扩展点（OCP）但不提前实现（YAGNI） |

**判断标准**：问自己"如果需求变了，改起来痛不痛？"痛 → 违反了重要的原则；不痛 → 当前设计够用。

## 反模式速查

| 反模式 | 违反的原则 | 生活比喻 | 信号 | 预防方式 |
|--------|-----------|---------|------|---------|
| God Object（万能管家） | SRP | 一个人既做饭又洗碗又买菜又修水管 | 一个类/函数做所有事 | 按职责拆分 |
| Shotgun Surgery（散弹枪手术） | SRP | 改一个灯泡要动天花板、地板和墙壁 | 改一个需求要改 5 个文件 | 把相关逻辑集中 |
| Hard-Coded Dispatch（硬编码分发） | OCP | 每来一个新客人就要重新装修客厅 | if type == "A" elif type == "B" | 用注册/插件机制 |
| Copy-Paste Programming（复制粘贴编程） | DRY | 三个房间各装一个热水器 | 相似代码块超过 3 处 | 提取公共逻辑 |
| Feature Envy（羡慕别人） | SRP | 厨师总在用客厅的电视 | 函数总在操作另一个模块的数据 | 把逻辑移到数据所在模块 |
| Middle Man（传话筒） | LoD | 一个人只负责把话传给另一个人 | 模块只是转发调用，没有自己的逻辑 | 去掉中间层或增加价值 |
| Primitive Obsession（原始类型偏执） | SRP | 用一堆散落的数字和文字描述一个人 | 用 dict/str 代替有行为的对象 | 创建专用数据类 |
| 上下文无限增长 | SSOT | 记忆力无限的大脑最终会崩溃 | 对话越来越慢越来越贵 | 设计时规定最大存储量 |
| 错误被吞掉 | SoC | 出了问题没人知道 | 操作没有返回成功/失败状态 | 每个操作必须返回状态 |
| 状态到处修改 | SSOT | 多个人同时改同一份文件 | 多个模块都能修改同一状态 | 只有指定模块可以修改状态 |
